#### 1. Introduction
1. Summary
   현대 사회에서 1인 가구가 증가하고 바쁜 일상이 지속됨에 따라, 효율적인 식재료 관리는 가계 경제와 건강 관리에 있어 필수적인 요소가 되었다.
   그러나 많은 사용자들이 냉장고 속 식재료의 유통기한을 제때 파악하지 못해 음식을 폐기하거나, 중복 구매를 하는 불편함을 겪고 있다.
   이에 직관적이고 스마트한 식재료 관리를 통해 음식물 쓰레기를 줄이고 소비 효율을 극대화하고자 만든 서비스이다.
2. Introduce "Save-My-Fridge"
   이번에 설계 및 제작하게 된 앱 "Save-My-Fridge"는 모바일 환경중심의 스마트 냉장고 관리 시스템이다.
   해당 시스템은 단순히 식재료의 수량을 기록하는 것을 넘어, 카테고리별 분류 및 보관위치(냉장/냉동) 지정을 통해 직관적인 인벤토리 환경을 제공한다.
   또한, 유통기한이 임박한 재료와 이미 만료된 재료를 시스템이 자동 판별하여 시각적인 경고로 알려주는 기능을 포함한다.
   사용자는 개별 식재료에 메모를 추가하거나 실시간 검색을 활용하여 냉장고 내부 상태를 완벽하게 통제하고 관리할 수 있다.
3. Goal
   이번 Design 보고서에서 앞서 정의한 유스케이스를 바탕으로 시스템의 동적 흐름을 명확히 하기 위해 Sequence diagram을 작성하고 상세동작과정을 소개한다.
   또한, 시스템을 구성하는 핵심 도메인과 데이터 구조를 구체화한 Class Diagram을 설계하여 각 클래스의 역할과 속성, 메서드 관계를 설명한다.
   더해서 앱의 생명 주기와 화면 전환 흐름을  한눈에 파악할 수 있도록 완성된 상태의 StateMachine Diagram을 그려 시스템이 가질 수 있는 모든 상태 변호를 시각화하여 보여줄 것이다.

#### 3. Sequence Diagram
1) Login
<img width="700" height="700" alt="1" src="https://github.com/user-attachments/assets/3873c7cc-17ed-4168-84b2-d288e4618960" />

Login Use case 에서의 Sequence Diagram. LoginController 클래스가 핵심 기능들을 제어하게 될 Controller 클래스이다. 사용자들이 로그인 화면에서 입력한 이메일과 비밀번호를 기반으로 loginCheck 메소드가 실행되면, 시스템은 CombinedFragment1(alt) 조건절을 통해 입력값의 적합성을 가장 먼저 평가한다. 이메일이나 비밀번호가 누락되었거나 일치하는 회원이 없을 경우 상위 Control 인터페이스의 startToast 메소드를 통해 경고 피드백을 출력한다. 모든 인증 조건을 통과한 정상 회원일 경우 [else] 절로 진입하여 User 데이터 객체로부터 getEmail() 메소드를 통해 정보를 호출하고 세션을 유지하게 된다. 최종적으로 로그인 성공 안내를 띄운 후 메인 대시보드 화면으로 안전하게 진입(changePage)하며 로직이 종료된다.

2) Register Ingredient
<img width="700" height="700" alt="2" src="https://github.com/user-attachments/assets/0df65b47-db19-40af-88d5-15d9d0ab5b48" />

재료 등록 Use case 에서의 Sequence Diagram. IngredientFormController 클래스가 등록 화면의 사용자 입력을 받아 제어하는 Controller 역할을 수행한다. 사용자가 식재료 입력 폼에 이름, 카테고리, 보관 위치, 유통기한 등의 정보를 타이핑한 후 등록 버튼을 누르면 registerIngredient() 메소드가 작동한다. 시스템은 CombinedFragment1(alt) 조건절을 통해 데이터 무결성을 먼저 체크하며, 필수 항목인 식재료 명칭이나 유통기한 날짜 정보가 누락되었을 경우 Control 인터페이스의 startToast 메소드를 호출해 사용자에게 경고 알림을 출력한다. 정상적으로 기입된 [else] 상태일 경우, 가공된 식재료 데이터 객체를 FridgeDatabase의 saveToDB(item) 메소드로 전달하여 로컬 저장소에 영구 기록한다. 이때 데이터베이스 파일에 기록이 완료되는 시점에서 UC #5(전체 목록 조회) 과정을 거쳐 최신 재고 상태를 화면에 리로드해야 하므로 해당 구간에 ref(Reference) 프래그먼트를 배치하여 타 시퀀스를 재사용하도록 명시하였다. 최종 성공 피드백을 받으면 등록 완료 토스트 메시지를 보여주고 메인 화면으로 전환하며 로직이 종료된다.

+8) Input/Edit Expiration Date

유통기한 입력/수정 유스케이스는 식재료 객체를 새로 생성하는 '2) 재료 등록' 과정 및 기존 객체의 속성을 변경하는 '4) 재료 수량 수정' 시퀀스 다이어그램의 제어 아키텍처를 그대로 공유 및 재사용한다. 
최초 등록 시에는 IngredientFormController를 통해 날짜 데이터가 입력되어 무결성 검증을 거쳐 FridgeDatabase에 저장되며, 상세 화면에서 유통기한을 수정할 때는 4번 수량 수정 흐름과 동일하게 대상 Ingredient 객체의 setExpirationDate() 메소드를 트리거한 후 saveToDB() 트랜잭션을 수립하게 된다. 따라서 시스템 내의 컴포넌트 호출 메커니즘과 파일 입출력 트랜잭션 흐름이 완벽히 일치하므로 중복 작성을 방지하기 위해 해당 다이어그램들을 통합 재사용하도록 처리한다.

+12) Set Storage Location

보관 위치 설정 유스케이스는 사용자가 식재료를 신규 등록하거나 저장 상태를 변경할 때 '냉장' 혹은 '냉동' 보관실을 지정하는 기능이다. 본 기능은 독립적인 시퀀스를 가지지 않고 '2) 재료 등록' 시퀀스 다이어그램 내부의 '1 : 식재료 정보 입력' 단계에서 입력 폼 데이터(storageType)의 속성 중 일부로 함께 바인딩되어 시스템으로 전달된다. 
컨트롤러 인터페이스를 거쳐 최종적으로 FridgeDatabase의 파일 데이터로 영구 기록되는 데이터 트랜잭션 흐름 및 생명주기가 완전히 일치하므로, 불필요한 도면의 중복 생성을 지양하고 '2) 재료 등록' 시퀀스 다이어그램을 공동 재사용하도록 명시하여 전체 설계 문서 간의 연계성 및 정합성을 완성하였다.

3) Delete Ingredient
<img width="700" height="650" alt="스크린샷 2026-06-04 161501" src="https://github.com/user-attachments/assets/57dca5c9-2db0-4ebf-869c-e8e7383f2617" />

재료 삭제 Use case 에서의 Sequence Diagram. 사용자가 식재료 상세 보기 화면에서 더 이상 필요 없거나 소비가 완료된 특정 식재료를 폐기하기 위해 삭제 버튼을 누를 때 작동하는 제어 흐름이다. 삭제 이벤트가 발생하면 IngredientFormController 클래스가 이를 가동하여 deleteIngredient() 메소드를 실행한다. 컨트롤러는 선택된 식재료의 고유 식별자(id)를 바인딩하여 FridgeDatabase의 removeFromDB(id) 메소드를 호출하고 데이터베이스 내에서 해당 식재료 레코드를 물리적으로 영구 파기하도록 요청한다. 삭제 처리가 진행되는 시점에서 메인 화면의 재고 목록에서도 해당 데이터가 즉각 제외되어야 하므로, 중복 설계를 배제하기 위해 해당 구간에 ref(Reference) 프래그먼트를 배치하여 UC #5(전체 목록 조회) 과정을 재사용하도록 설계하였다. 데이터베이스로부터 제거 완료 피드백(success)을 수신하면, 최종적으로 Control 인터페이스를 통해 유저의 화면을 안전하게 메인 목록 창으로 전환(changePage)시키고 삭제 완료 알림을 화면에 띄우며 종료된다.

4) Edit Quantity
<img width="700" height="300" alt="4" src="https://github.com/user-attachments/assets/af71c1e1-3801-41e1-8ba1-51b34d969952" />

재료 수량 수정 Use case 에서의 Sequence Diagram. 유저가 냉장고 대시보드나 식재료 상세 보기 화면에서 특정 재료의 수량을 증감하기 위해 +, - 버튼을 클릭할 때 작동하는 시스템의 상호작용 흐름이다. 사용자가 버튼을 클릭하면 IngredientFormController 클래스가 이를 감지하여 editQuantity(amount) 메소드를 호출하고 비즈니스 로직을 가동한다. 컨트롤러는 내부 연산을 통해 기존 수량에 증감치를 더해 새로운 수량(newQty)을 계산한 뒤, 대상 데이터 모델인 Ingredient 객체의 setQuantity(newQty) 메소드를 호출하여 객체의 상태 값을 실시간으로 변경한다. 수정이 완료되면 변경된 식재료 객체를 FridgeDatabase의 saveToDB(selectedItem) 메소드로 전달하여 로컬 저장소 파일에 물리적으로 영구 기록하도록 요청한다. 이때 수량이 변경되었으므로 화면 대시보드의 목록 정보 역시 최신화되어야 하므로, 중복 설계를 방지하기 위해 해당 구간에 ref 프래그먼트를 배치하여 UC #5(전체 목록 조회) 과정을 재사용하도록 명시하였다. 데이터베이스로부터 성공 피드백을 수신하면 최종적으로 사용자 화면 하단에 수량 변경 완료 알림을 안내하며 종료된다.

5) Show Inventory Main
<img width="700" height="600" alt="5" src="https://github.com/user-attachments/assets/c25ee152-2581-496e-97e2-715b36868e57" />

전체 목록 조회 Use case 에서의 Sequence Diagram. 사용자가 앱 로그인을 완료하여 메인 대시보드 화면에 진입하거나 화면을 새로고침할 때 수행되는 시스템의 전반적인 제어 흐름이다. InventoryMainController는 화면 활성화 시점에 가장 먼저 FridgeDatabase의 loadFromDB() 메소드를 호출하여 로컬 저장소에 안전하게 보관되어 있는 전체 식재료 데이터 리스트(ingredientList)를 동적으로 불러온다. 데이터를 성공적으로 반환받은 후, 자체 비즈니스 로직인 checkExpirationRules() 메소드를 실행하여 로드된 모든 식재료 데이터의 유통기한 D-day 상태를 전수 검사한다. 
이때 시스템은 CombinedFragment1(alt) 조건절을 통해 총 세 가지 케이스로 분기하여 유저에게 다르게 반응한다. 첫째, 요구사항 정의서의 '유통기한 지난 재료 표기(폐기)' 조건에 따라 유통기한이 이미 지나버린 식재료(D-day <= 0)가 적발될 경우, 상위 Control 인터페이스의 makeNotice 메소드를 트리거하여 강한 경고 팝업창을 발생시킨다. 둘째, '유통기한 임박 표시(3일 이내)' 조건에 따라 유통기한이 3일 이내로 남은 임박 식재료(0 < D-day <= 3)가 존재할 경우, startToast 메소드를 통해 유저에게 소비를 촉구하는 토스트 안내를 띄운다. 마지막으로 아무런 이상 항목이 없는 안전한 [else] 상태일 경우, 대시보드 화면에 리스트 데이터를 정상 바인딩하여 표출(showAllList)하며 시퀀스가 종료된다.

+9) Imminent Expiration Notification

유통기한 임박 표시 유스케이스는 독립적인 시퀀스를 가지지 않고, 시스템이 구동되어 메인 대시보드 인벤토리를 호출하는 시점에 백그라운드 스레드 연산을 통해 유기적으로 통합 실행되는 기능이다. 
따라서 본 기능은 앞서 상세 설계한 '5) 전체 목록 조회' 시퀀스 다이어그램 내부의 CombinedFragment1(alt) 조건 분기 구조에 완벽히 매핑되어 표현되어 있다. InventoryMainController가 checkExpirationRules() 메소드를 통해 전체 식재료의 시간 데이터를 검사할 때, 유통기한이 3일 이내로 남은 임박 대상이 적발되면 [0 < D-day <= 3] 절이 수행되어 상위 Control 인터페이스의 startToast() 피드백 알림이 트리거된다. 결과적으로 해당 메커니즘은 전체 목록 조회 흐름에 포함되어 자동 작동하므로 '5) 전체 목록 조회' 도면을 공동 재사용한다.

+10) Expired Ingredient Notification

유통기한 지난 재료 표기 유스케이스 역시 '5) 전체 목록 조회' 시퀀스 다이어그램 내부의 CombinedFragment1(alt) 조건 분기 구조에 포함되어 작동하는 연산 흐름이다. 
InventoryMainController가 식재료 리스트를 동적으로 읽어온 직후 checkExpirationRules() 메소드를 실행하여 유통기한이 이미 지나버린 폐기 대상 식재료(D-day <= 0)를 적발해내면, 최상위 우선순위 조건인 [D-day <= 0] 절이 수행되어 사용자가 앱에 접속하자마자 폐기 경고 창을 확인할 수 있도록 Control.makeNotice() 강제 경고 팝업창을 발생시킨다. 대시보드 인프라 데이터 로드 시 함께 처리되는 비즈니스 아키텍처이므로 '5) 전체 목록 조회' 도면을 공동 재사용하여 정합성을 수립한다.

6) Search By Name
<img width="700" height="600" alt="6" src="https://github.com/user-attachments/assets/fe8c555e-6b9d-426a-9eb8-79b554b0f0e6" />

이름 검색 Use case 에서의 Sequence Diagram. 사용자가 메인 대시보드 화면 상단에 위치한 검색창에 식재료 명칭을 타이핑할 때 작동하는 동적 조회 제어 흐름이다. 카테고리 필터 Use case 에서의 Sequence Diagram. 사용자가 메인 화면 상단에 나열된 카테고리 필터 탭 뷰(채소, 육류, 유제품 등)를 터치하여 특정 분류 항목만 모아보고자 할 때의 상호작용 흐름이다. 사용자가 특정 탭을 선택하면 SearchFilterController가 해당 카테고리 명칭을 인자값으로 포맷팅하여 InventoryMainController의 filterByCategory(category) 메소드를 호출한다. 메인 컨트롤러는 내부 비즈니스 로직을 통해 냉장고 재고 리스트에 존재하는 모든 식재료 객체들의 category 속성 값을 전수 조사하며 탭 메뉴와 일치하는 데이터 세트만 선별하는 정렬 연산을 처리한다. 조율이 완료되면 매칭된 카테고리 식재료 그룹만을 바인딩하여 showAllList() 메소드를 실행함으로써, UI 대시보드 화면에 해당 분류 영역의 재고 데이터만 동적으로 압축하여 유저에게 보여주며 시퀀스가 종료된다.

메모 추가 Use case 에서의 Sequence Diagram이다. 사용자가 식재료 상세 정보 화면에서 해당 재료의 특이사항이나 보관 팁 내용을 기록하고 저장하고자 할 때 작동하는 시스템 제어 흐름이다. 유저가 상세 페이지의 메모 폼에 텍스트를 타이핑하면 IngredientFormController 클래스가 이를 수집하여 addMemo(text) 메소드를 호출한다. 컨트롤러는 입력된 문자열을 가공한 뒤, 해당 식재료 데이터 모델 인스턴스인 Ingredient 객체의 setMemo(text) 메소드를 실행하여 객체 내부의 memo 속성 값을 실시간으로 갱신한다.

#### 4. State Machine Diagram

<img width="700" height="600" alt="스크린샷 2026-06-04 171317" src="https://github.com/user-attachments/assets/cb65cf09-79a8-4529-93ad-fca403666bf5" />

본 다이어그램은 'Save-My-Fridge' 시스템의 핵심 타겟 객체인 '식재료(Ingredient)'의 생명주기(Lifecycle)에 따른 상태 전이를 표현한 단일 상태 다이어그램(State Diagram)이다. 교수님이 제시한 ATM 예시의 중첩 프레임(Composite State) 표준 표기법을 준수하여, 요구사항 정의서에 명시된 기능적 조건들을 하나의 흐름으로 통합 설계하였다.

1. 초기 진입 및 보관 설정
초기 데이터가 없는 '미등록(Unregistered)' 상태에서 유저가 재료 등록을 수행하면, 객체는 '냉장고 보관 중(In Fridge)'이라는 거대한 복합 상태 프레임 내부로 진입한다. 진입과 동시에 'entry / 유통기한 D-day 계산' 액션이 수행되며, 'do / 보관 위치 설정(냉장/냉동)' 상태가 기본적으로 바인딩된다.

2. 복합 상태(In Fridge) 내부의 세부 전이 연산
- 정상 보관(Safe): 유통기한이 안정적인 상태이다. 유저가 메인 화면에서 '이름 검색'을 수행하거나 '카테고리 필터' 탭을 터치하여 목록을 거르는 액션은 보관 상태를 해치지 않으므로 내부에서 순환하는 흐름으로 제어된다. 또한 '수량 수정(editQuantity)' 역시 객체 자체의 수량 속성만 변경될 뿐 보관 상태는 유지되므로 자기 재진입(Self-Transition) 선으로 표현되었다.
- 유통기한 임박(Imminent): 시스템 백그라운드 연산(checkExpirationRules)을 통해 남은 기한이 3일 이내(0 < D-day <= 3)로 좁혀지면 '유통기한 임박 표시' 상태로 자동 전이된다.
- 유통기한 만료(Expired): 기한이 완전히 경과(D-day <= 0)되면 '유통기한 지난 재료 표기(폐기)' 상태로 전이되며, 최종 마감 상태에 도달한다.

3. 전역 상태 탈출 및 파기
식재료 데이터가 복합 상태 내부의 어느 지점(정상, 임박, 만료)에 머물러 있든 상관없이, 유저가 '재료 삭제' 버튼을 클릭하여 deleteIngredient()를 가동하면 복합 프레임의 테두리 경계선을 탈출하여 즉각 '파기(Deleted)' 상태로 전이된 후, 최종 소멸(Final State)하며 전체 생명주기가 완전 종료된다.
